# [Decision] 데이터베이스 선택: PostgreSQL vs MongoDB

> Nova Engineering — Decision Record (경량 ADR)
> 날짜: 2026-03-18
> 상태: 승인됨

---

## Context (배경)

ZipPit 프로젝트의 백엔드 데이터베이스를 선정해야 한다. 부동산 금융 매칭 플랫폼의 특성상 사용자, 매물, 금융 상품, 매칭 결과 등 다양한 도메인 데이터를 저장하고 복잡한 관계를 조회해야 한다.

현재 상태:
- NestJS 백엔드 프레임워크 결정 완료
- TypeORM 또는 Prisma를 ORM으로 사용할 예정
- 초기 MVP 단계이나, 6개월 내 정식 서비스 출시 목표
- 팀원 4명 중 3명이 관계형 DB 경험, 1명이 MongoDB 경험 보유

## Problem (문제)

부동산 금융 매칭 서비스에 적합한 데이터베이스를 PostgreSQL과 MongoDB 중 선택해야 한다.

핵심 고려사항:
- 매물-금융상품 간 다대다 관계 및 복잡한 조인 쿼리
- 금융 거래 데이터의 정합성(ACID) 보장
- 매물 검색 시 위치 기반 쿼리 필요 (GIS)
- 금융 상품 스펙이 상품 유형마다 다른 스키마를 가질 수 있음

## Decision (결정)

**PostgreSQL을 메인 데이터베이스로 채택한다.**

선택 이유:
1. 부동산 금융 매칭의 핵심은 매물-금융상품-사용자 간 관계 조회이며, 관계형 DB가 이에 최적화되어 있다
2. 금융 데이터 특성상 ACID 트랜잭션이 필수적이며, PostgreSQL이 이를 네이티브로 보장한다
3. PostGIS 확장으로 위치 기반 검색을 별도 인프라 없이 지원할 수 있다
4. 금융 상품의 유동적 스키마는 PostgreSQL의 JSONB 컬럼으로 충분히 수용 가능하다

### 대안 비교

| 기준 | PostgreSQL | MongoDB | 비고 |
|------|-----------|---------|------|
| 관계형 쿼리 | 네이티브 JOIN, 서브쿼리 | $lookup (제한적) | 매물-금융상품 매칭에 핵심 |
| 트랜잭션 (ACID) | 완전 지원 | 4.0+ 지원하나 성능 오버헤드 | 금융 데이터 필수 요건 |
| 위치 기반 검색 | PostGIS (성숙, 고성능) | GeoJSON (기본 지원) | PostGIS가 기능/성능 우위 |
| 유연한 스키마 | JSONB 컬럼으로 부분 지원 | 네이티브 지원 (강점) | 금융상품 스펙 다양성 대응 |
| ORM 지원 | TypeORM/Prisma 완전 지원 | Mongoose (별도 생태계) | NestJS 통합 용이성 |
| 팀 역량 | 3/4명 경험 보유 | 1/4명 경험 보유 | 학습 비용 고려 |
| 운영 비용 | AWS RDS 지원, NCP Cloud DB 지원 | Atlas 또는 자체 운영 | 멀티클라우드 호환 |
| 결론 | **채택** | 기각 (관계형 쿼리 약점, 팀 역량 부족) | |

## Consequences (결과)

### 긍정적
- 매물-금융상품 매칭 로직을 SQL JOIN으로 효율적으로 구현 가능
- 금융 거래의 데이터 무결성을 트랜잭션으로 확실히 보장
- PostGIS로 "내 주변 매물 검색" 기능을 고성능으로 제공
- 팀 대부분이 익숙하여 초기 개발 속도 확보
- JSONB로 금융 상품별 유동적 속성도 유연하게 저장 가능

### 부정적
- 금융 상품 스키마가 극도로 다양해질 경우 JSONB 쿼리 성능이 저하될 수 있음
- 수평 확장(샤딩)이 MongoDB 대비 복잡함 (대규모 트래픽 시 고려 필요)
- 비정형 로그 데이터 저장에는 오버스펙일 수 있음

### 주의사항
- MAU 10만 이상 또는 매물 100만 건 이상 시 읽기 성능 최적화 필요 (Read Replica 도입 검토)
- 금융 상품 유형이 50개 이상으로 증가하면 JSONB 인덱싱 전략 재검토
- 비정형 데이터(사용자 행동 로그 등)가 대량 발생하면 별도 저장소(예: Elasticsearch) 추가 검토

합의율: 85% → 판정: 사람 판단

**차이점 분석**:
- 3개 AI 모두 PostgreSQL을 메인 DB로 추천하는 데 합의 (핵심 합의 100%)
- GPT가 Polyglot Persistence(보조 DB로 MongoDB 병행)를 제안한 부분이 차이점
- 합의율이 85%인 이유: GPT의 조건부 합의 (보조 DB 제안은 현 단계에서는 과한 설계라는 점에서 의견 차이)

**최종 결정 (사람 판단)**:
MVP 단계에서는 PostgreSQL 단독으로 진행한다. GPT의 Polyglot Persistence 제안은 유효하나 현재 시점에서는 복잡도 증가 대비 이점이 불분명하다. 매물 100만 건 돌파 시점에 데이터 저장 전략을 재검토하기로 한다.
