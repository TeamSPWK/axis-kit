## Context (배경)

### 현재 상태
- ZipPit 플랫폼이 MVP 단계로, 현재는 인증 없이 모든 API가 공개 상태
- 프론트엔드(Next.js)와 백엔드(NestJS) 분리 아키텍처로 운영 중
- 사용자 데이터 테이블은 존재하나 비밀번호/토큰 관련 필드 없음

### 왜 필요한가
- 베타 출시 전 사용자 식별 및 접근 제어가 필수 (출시 예정: 2026-04-15)
- 부동산 금융 매칭 특성상 개인정보 보호 규정(개인정보보호법) 준수 필요
- 파트너사(금융기관) 연동 시 인증된 사용자 기반의 API 호출이 요구됨

## Problem (문제 정의)

### 핵심 문제
인증 시스템이 없어 사용자를 식별할 수 없고, API를 보호할 수 없다.

### MECE 분해

| # | 문제 영역 | 설명 | 영향도 |
|---|----------|------|--------|
| 1 | 신원 확인 | 회원가입/로그인으로 사용자 신원을 확인하는 메커니즘 부재 | 높음 |
| 2 | 세션 유지 | 로그인 후 상태를 유지하는 토큰/세션 관리가 없음 | 높음 |
| 3 | API 보호 | 인증되지 않은 요청으로부터 보호되는 엔드포인트가 없음 | 높음 |
| 4 | 비밀번호 보안 | 비밀번호 저장, 해싱, 재설정 등 보안 처리 로직 부재 | 중간 |
| 5 | 소셜 로그인 | 카카오/네이버 등 소셜 로그인 지원 요구사항 | 낮음 |

### 제약 조건
- **기술적 제약**: 기존 NestJS + PostgreSQL 스택 위에 구현해야 함
- **시간 제약**: 베타 출시까지 3주 (소셜 로그인은 2차에 수용 가능)
- **비즈니스 제약**: 이메일 기반 가입 필수, 핸드폰 인증은 후순위

## Solution (해결 방안)

### 선택한 방안
JWT(JSON Web Token) 기반 인증 시스템 구현. Access Token(15분) + Refresh Token(7일) 이중 토큰 전략으로 보안과 사용자 경험을 균형 있게 확보한다.

### 대안 비교

| 기준 | JWT 이중 토큰 | 서버 세션 (Redis) | 외부 인증 서비스 (Auth0) |
|------|-------------|-------------------|------------------------|
| 구현 복잡도 | 중간 — 직접 구현 필요 | 중간 — Redis 인프라 추가 필요 | 낮음 — SDK 연동 |
| 확장성 | 높음 — Stateless, 수평 확장 용이 | 중간 — Redis 클러스터 필요 | 높음 — 관리형 |
| 비용 | 낮음 — 인프라 추가 없음 | 중간 — Redis 서버 비용 | 높음 — MAU당 과금 |
| 커스터마이징 | 높음 — 완전 제어 가능 | 높음 — 완전 제어 가능 | 낮음 — 공급자 의존 |
| 오프라인 검증 | 가능 — 서명 검증만으로 충분 | 불가 — 매 요청마다 Redis 조회 | 가능 — JWT 기반 |
| 선택 | **채택** | 기각 (Redis 인프라 부담) | 기각 (비용 + 벤더 종속) |

### 구현 범위
- [ ] 사용자 테이블에 password_hash, refresh_token 필드 추가 (마이그레이션)
- [ ] 회원가입 API (POST /auth/register) — bcrypt 해싱
- [ ] 로그인 API (POST /auth/login) — JWT 발급
- [ ] 토큰 갱신 API (POST /auth/refresh) — Refresh Token 회전
- [ ] 로그아웃 API (POST /auth/logout) — Refresh Token 무효화
- [ ] AuthGuard 미들웨어 — 보호된 라우트에 적용
- [ ] 비밀번호 재설정 (POST /auth/reset-password) — 이메일 토큰 방식

### 검증 기준
- 회원가입 후 즉시 로그인이 가능하다
- Access Token 만료 시 Refresh Token으로 자동 갱신된다
- 유효하지 않은 토큰으로 보호된 API 접근 시 401 응답을 받는다
- 비밀번호는 DB에 평문으로 저장되지 않는다

| AI | 의견 요약 | 합의 |
|----|----------|------|
| Claude | JWT 이중 토큰 추천. Stateless 특성이 마이크로서비스 전환 시 유리. Refresh Token 회전 필수 강조. | O |
| GPT | JWT 추천하되, Refresh Token을 DB에 저장하여 강제 로그아웃 지원 권장. 순수 Stateless보다 현실적. | O |
| Gemini | JWT 동의. 단, Token Blacklist 없이는 Access Token 탈취 시 만료까지 차단 불가한 점 유의. 짧은 만료시간(15분 이하)으로 완화 가능. | O |

합의율: 95% → 판정: 자동 채택

**합의 요약**: 3개 AI 모두 JWT 이중 토큰 방식에 동의. 공통 권장사항으로 (1) Refresh Token DB 저장, (2) Access Token 만료시간 15분 이하, (3) Refresh Token 회전(rotation) 적용을 반영하기로 결정.
