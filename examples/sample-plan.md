- **사용자 요구**: 베타 테스터 피드백에서 "내 매칭 이력을 다시 볼 수 없다"는 불만 다수

## Problem (문제 정의)

### 핵심 문제
사용자를 식별하고 권한을 관리하는 인증 시스템이 없어, 개인화 서비스 제공과 데이터 보호가 불가능하다.

### MECE 분해

| # | 문제 영역 | 설명 | 영향도 |
|---|----------|------|--------|
| 1 | 인증 (Authentication) | 사용자가 누구인지 확인하는 메커니즘이 없음. 이메일/비밀번호 또는 소셜 로그인 필요 | 높음 |
| 2 | 인가 (Authorization) | 인증된 사용자의 역할(일반/관리자)에 따른 접근 제어가 없음 | 높음 |
| 3 | 세션 관리 | 로그인 상태를 유지하고 만료시키는 메커니즘이 없음. 보안과 UX 모두에 영향 | 중간 |
| 4 | 계정 관리 | 비밀번호 재설정, 이메일 변경, 계정 삭제 등 부가 기능 부재 | 낮음 |

### 제약 조건
- **기술적**: 기존 Express.js 백엔드에 통합해야 함. 별도 인증 서버 구축 리소스 없음
- **시간**: 2주 내 MVP 인증 완성 필요 (베타 2차 출시 일정)
- **비즈니스**: 소셜 로그인(카카오)은 1차 출시에 반드시 포함해야 함 (국내 서비스 특성)

## Solution (해결 방안)

### 선택한 방안
JWT(JSON Web Token) 기반 인증 + Refresh Token 로테이션 방식을 채택한다. Access Token은 15분 만료, Refresh Token은 7일 만료로 설정하고, HttpOnly 쿠키에 저장한다.

### 대안 비교

| 기준 | JWT + Refresh Token | 서버 세션 (Redis) | 외부 서비스 (Auth0) |
|------|-------------------|-----------------|-------------------|
| 구현 복잡도 | 중간 — 토큰 발급/검증 직접 구현 | 중간 — Redis 인프라 추가 필요 | 낮음 — SDK 연동만 |
| 확장성 | 높음 — Stateless, 서버 증설 용이 | 중간 — Redis 클러스터 필요 | 높음 — 위임 |
| 비용 | 무료 | Redis 인프라 비용 월 ~3만원 | MAU 1만 초과 시 월 $23+ |
| 커스터마이징 | 자유로움 | 자유로움 | 제한적 |
| 보안 | 토큰 탈취 시 만료까지 유효 (Refresh 로테이션으로 완화) | 서버에서 즉시 무효화 가능 | 전문 보안팀 관리 |
| 선택 | **채택** | 기각 (인프라 추가 부담) | 기각 (비용, 카카오 커스텀 제한) |

### 구현 범위
- [x] 이메일/비밀번호 회원가입 및 로그인
- [x] JWT Access Token + Refresh Token 발급
- [x] 카카오 소셜 로그인 (OAuth 2.0)
- [x] 역할 기반 접근 제어 (RBAC: user, admin)
- [ ] 비밀번호 재설정 (2차 배포)
- [ ] 네이버/구글 소셜 로그인 (2차 배포)

### 검증 기준
- 회원가입 후 로그인하면 Access Token이 발급되고, 보호된 API에 접근 가능해야 한다
- Access Token 만료 후 Refresh Token으로 자동 갱신이 되어야 한다
- 카카오 로그인으로 신규 가입과 기존 계정 연결이 모두 동작해야 한다
- admin 역할이 아닌 사용자가 관리자 API에 접근하면 403이 반환되어야 한다
